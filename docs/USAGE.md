# PMI — Руководство по использованию

**PMI (Parallel Motion Interpolate)** — утилита для повышения частоты кадров видео через оптический поток (алгоритм `minterpolate`). Видео разбивается на сегменты, каждый обрабатывается параллельно, затем сегменты склеиваются. Аудио и субтитры копируются без перекодирования.

## 1. Быстрый старт

```bash
# Повысить FPS фильма до 60 с настройками по умолчанию
./PMI -i film.mkv -o film_60fps.mkv --fps=60

# Минимальная команда (входной файл как позиционный аргумент)
./PMI film.mkv --fps=60
# → создаст output.mkv
```

## 2. Полный справочник параметров

### Основные параметры

| Параметр | Умолчание | Описание |
|---|---|---|
| `-i`, `--input=FILE` | `input.mkv` | Входной видеофайл |
| `-o`, `--output=FILE` | `output.mkv` | Выходной видеофайл |
| `--fps=N` | `60` | Целевая частота кадров (1–240) |
| `-j`, `--jobs=N` | авто (= CPU) | Число параллельных потоков |

### Параметры интерполяции (`minterpolate`)

| Параметр | Умолчание | Варианты | Описание |
|---|---|---|---|
| `--mi-mode=MODE` | `mci` | `mci`, `blend`, `dup` | Алгоритм интерполяции |
| `--mc-mode=MODE` | `aobmc` | `aobmc`, `obmc` | Алгоритм компенсации движения |
| `--me-mode=MODE` | `bidir` | `bidir`, `bilat` | Алгоритм оценки движения |
| `--vsbmc=N` | `1` | `0`, `1` | Variable-size block motion compensation |

### Параметры кодирования (x264)

| Параметр | Умолчание | Описание |
|---|---|---|
| `--preset=NAME` | `slow` | Скорость/качество: `ultrafast` → `veryslow` |
| `--crf=N` | `18` | Качество: 0 (лучше) — 51 (хуже), 18–23 оптимально |

### Служебные параметры

| Параметр | Умолчание | Описание |
|---|---|---|
| `--temp-dir=DIR` | `.pmi_tmp_*` | Папка для временных сегментов |
| `--keep-temp` | выкл. | Не удалять временные сегменты после завершения |
| `-v`, `--verbose` | выкл. | Подробный вывод FFmpeg (`AV_LOG_INFO`) |
| `-h`, `--help` | — | Справка |

## 3. Примеры запуска

### Базовые сценарии

```bash
# 24 → 60 fps, стандартные настройки
./PMI -i film.mkv -o film_60fps.mkv --fps=60

# 24 → 120 fps
./PMI -i film.mkv -o film_120fps.mkv --fps=120

# 25 → 90 fps (спорт, анимация)
./PMI -i cartoon.mp4 -o cartoon_90fps.mkv --fps=90

# Входной файл как первый аргумент (без -i)
./PMI film.mkv --fps=60 -o film_60fps.mkv
```

### Наивысшее качество интерполяции

Максимальное качество — самый медленный и точный режим. Рекомендуется для кинематографического материала, где важна точность воспроизведения движения:

```bash
./PMI \
  -i film.mkv \
  -o film_60fps_best.mkv \
  --fps=60 \
  --mi-mode=mci \
  --mc-mode=aobmc \
  --me-mode=bilat \
  --vsbmc=1 \
  --preset=veryslow \
  --crf=16 \
  -j 1
```

**Почему `-j 1`:** при наивысшем качестве параллельная обработка может давать незначительные артефакты на стыках сегментов (minterpolate теряет контекст между сегментами). Один поток обрабатывает всё видео целиком.

**Почему `--me-mode=bilat`:** билатеральный алгоритм оценки движения точнее `bidir` для сцен с неравномерным движением, но значительно медленнее.

### Высокое качество, умеренная скорость

Хороший баланс для большинства фильмов:

```bash
./PMI \
  -i film.mkv \
  -o film_60fps_hq.mkv \
  --fps=60 \
  --mi-mode=mci \
  --mc-mode=aobmc \
  --me-mode=bidir \
  --vsbmc=1 \
  --preset=slow \
  --crf=18
```

### Быстрая обработка (предпросмотр)

Для проверки результата перед финальной обработкой:

```bash
./PMI \
  -i film.mkv \
  -o film_60fps_preview.mkv \
  --fps=60 \
  --mi-mode=blend \
  --preset=ultrafast \
  --crf=26
```

`blend` вместо `mci` работает в ~5 раз быстрее, но просто смешивает соседние кадры без оптического потока — результат менее чёткий.

### Анимация и мультфильмы

Для анимации без сложного движения камеры. `dup` просто дублирует кадры без интерполяции — подходит если нужен только формально корректный FPS без плавности:

```bash
# Плавная интерполяция для аниме (осторожно: может давать артефакты на резких вспышках)
./PMI \
  -i anime.mkv \
  -o anime_60fps.mkv \
  --fps=60 \
  --mi-mode=mci \
  --mc-mode=obmc \
  --me-mode=bidir \
  --vsbmc=0 \
  --preset=slow \
  --crf=18

# Простое дублирование кадров (без артефактов, но без плавности)
./PMI -i anime.mkv -o anime_60fps_dup.mkv --fps=60 --mi-mode=dup
```

### Управление потоками

```bash
# Использовать все ядра (по умолчанию)
./PMI -i film.mkv -o out.mkv --fps=60

# Ограничить 4 потоками (оставить ядра для других задач)
./PMI -i film.mkv -o out.mkv --fps=60 -j 4

# Один поток — для наивысшего качества или диагностики
./PMI -i film.mkv -o out.mkv --fps=60 -j 1
```

### Сохранение временных файлов (отладка)

```bash
# Сохранить сегменты для анализа
./PMI \
  -i film.mkv \
  -o out.mkv \
  --fps=60 \
  --temp-dir=/tmp/pmi_segments \
  --keep-temp \
  -v

# После обработки сегменты доступны в /tmp/pmi_segments/
ls /tmp/pmi_segments/
```

### Разные входные форматы

```bash
# MKV
./PMI -i film.mkv -o out.mkv --fps=60

# MP4 (H.264, HEVC)
./PMI -i film.mp4 -o out.mkv --fps=60

# AVI
./PMI -i film.avi -o out.mkv --fps=60

# MPEG-TS (трансляции, записи эфира)
./PMI -i broadcast.ts -o out.mkv --fps=60

# FLV
./PMI -i video.flv -o out.mkv --fps=60
```

Выход всегда в MKV или MP4 (определяется расширением `-o`):
```bash
./PMI -i film.mkv -o out.mp4 --fps=60
```

## 4. Настройки качества интерполяции

### `--mi-mode` — алгоритм интерполяции

Главный параметр, влияющий на качество и скорость.

| Режим | Скорость | Качество | Описание |
|---|---|---|---|
| `mci` | медленно | ★★★★★ | Motion Compensated Interpolation — вычисляет векторы движения и синтезирует промежуточные кадры. Лучший результат для видео с плавным движением |
| `blend` | быстро | ★★★☆☆ | Смешение соседних кадров с весами. Быстро, но даёт «двоение» на быстром движении |
| `dup` | мгновенно | ★★☆☆☆ | Дублирование ближайшего кадра. Без артефактов, но и без плавности |

**Рекомендации по контенту:**

| Тип контента | Рекомендуемый режим |
|---|---|
| Игровое кино, сериалы | `mci` |
| Документальное видео | `mci` |
| Спорт с быстрым движением | `mci` + `bilat` |
| Аниме, мультфильмы | `mci` или `blend` |
| Слайдшоу, статичные сцены | `dup` |
| Предпросмотр/черновик | `blend` |

### `--mc-mode` — компенсация движения

Применяется только при `--mi-mode=mci`.

| Режим | Скорость | Качество | Описание |
|---|---|---|---|
| `aobmc` | медленнее | ★★★★★ | Adaptive Overlapped Block MC — адаптивный перекрывающийся алгоритм. Устраняет блочные артефакты на границах |
| `obmc` | быстрее | ★★★★☆ | Overlapped Block MC — базовый перекрывающийся алгоритм |

### `--me-mode` — оценка движения

Применяется только при `--mi-mode=mci`.

| Режим | Скорость | Качество | Описание |
|---|---|---|---|
| `bidir` | быстрее | ★★★★☆ | Двунаправленная оценка движения. Хорошо для большинства сцен |
| `bilat` | медленнее | ★★★★★ | Билатеральная оценка движения. Точнее для сцен с неоднородным движением (зум, панорама + объекты) |

### `--vsbmc` — блоки переменного размера

| Значение | Описание |
|---|---|
| `1` (вкл.) | Использовать блоки переменного размера. Лучше на мелких деталях, медленнее |
| `0` (выкл.) | Фиксированный размер блоков. Быстрее, менее точно на сложных сценах |

### Матрица качества

Суммарный рейтинг популярных комбинаций (★ = лучше):

| Комбинация | Качество | Скорость | Применение |
|---|---|---|---|
| `mci` + `aobmc` + `bilat` + `vsbmc=1` | ★★★★★ | ★☆☆☆☆ | Кино, архивация |
| `mci` + `aobmc` + `bidir` + `vsbmc=1` | ★★★★☆ | ★★★☆☆ | Повседневное использование |
| `mci` + `obmc` + `bidir` + `vsbmc=0` | ★★★☆☆ | ★★★★☆ | Быстрая обработка |
| `blend` + любые | ★★☆☆☆ | ★★★★★ | Предпросмотр |
| `dup` + любые | ★☆☆☆☆ | ★★★★★ | Только смена FPS без эффекта |

## 5. Настройки кодирования x264

### `--crf` — качество видео

CRF (Constant Rate Factor) управляет качеством выходного видео. Меньше = лучше качество, больше файл.

| CRF | Качество | Размер файла | Рекомендация |
|---|---|---|---|
| 0 | Lossless | Очень большой | Только для мастеринга |
| 16–17 | Визуально прозрачное | Большой | Архивация |
| 18 | Очень высокое | Большой | Значение по умолчанию |
| 20–22 | Высокое | Средний | Хранение и стриминг |
| 23–25 | Хорошее | Средний | Стриминг |
| 28+ | Заметные потери | Малый | Не рекомендуется |

> При мinterpolate синтезированные кадры уже содержат некоторую погрешность, поэтому CRF 18–20 достаточно — разница с CRF 16 на интерполированном материале практически незаметна.

### `--preset` — скорость кодирования x264

Preset влияет только на скорость и размер файла, не на визуальное качество при одинаковом CRF.

| Preset | Скорость | Сжатие | Применение |
|---|---|---|---|
| `ultrafast` | ★★★★★ | ★☆☆☆☆ | Предпросмотр |
| `superfast` | ★★★★☆ | ★★☆☆☆ | Черновик |
| `veryfast` | ★★★★☆ | ★★★☆☆ | Быстрое сохранение |
| `faster` | ★★★☆☆ | ★★★☆☆ | |
| `fast` | ★★★☆☆ | ★★★★☆ | |
| `medium` | ★★★☆☆ | ★★★★☆ | Баланс |
| `slow` | ★★☆☆☆ | ★★★★★ | По умолчанию, рекомендуется |
| `slower` | ★★☆☆☆ | ★★★★★ | |
| `veryslow` | ★☆☆☆☆ | ★★★★★ | Максимальное сжатие |

## 6. Параллелизм и производительность

### Как работает параллелизм

PMI делит входное видео на N равных сегментов (по числу потоков) и обрабатывает их одновременно. Каждый поток:
- декодирует свою часть исходника с seek
- читает дополнительно 2 секунды до начала сегмента (для контекста minterpolate)
- применяет фильтрграф minterpolate
- кодирует в x264
- записывает временный `.mkv`

После завершения всех потоков `concat` склеивает сегменты и мультиплексирует аудио/субтитры из оригинала.

### Ориентиры производительности

Пример на видео 1920×800, 24 fps → 60 fps, 8 ядер:

| Preset | mi-mode | me-mode | Время (18 сек видео) |
|---|---|---|---|
| `slow` + `mci` + `bidir` | mci | bidir | ~6 мин |
| `slow` + `mci` + `bilat` | mci | bilat | ~10–12 мин |
| `ultrafast` + `blend` | blend | — | ~30 сек |

Реальное время масштабируется линейно с длиной видео.

### Рекомендации по числу потоков

```bash
# Максимальная скорость — все ядра (по умолчанию)
./PMI -i film.mkv -o out.mkv --fps=60

# Оставить систему отзывчивой — половина ядер
./PMI -i film.mkv -o out.mkv --fps=60 -j $(( $(nproc) / 2 ))

# Один поток — наивысшее качество без артефактов стыков
./PMI -i film.mkv -o out.mkv --fps=60 -j 1
```

### Минимальная длина видео

PMI требует минимум **4 секунды** на сегмент (константа `MIN_SEG_DURATION`). Для 8-ядерной машины и 18-секундного видео будет создано только 4 сегмента, а не 8. Для более полного использования ядер нужны видео длиннее ~32 секунд.

| Кодек | Примечание |
|---|---|

### Выходные форматы

Выходной контейнер определяется расширением файла `-o`:

| Расширение | Контейнер |

Видеокодек выхода всегда **H.264 (libx264)**.

### Аудио и субтитры

Все аудио-дорожки и субтитры из исходника копируются без перекодирования:

| Тип | Поддерживаемые кодеки |

| Субтитры | ASS/SSA, SRT, SubRip, DVD sub, PGS (Blu-ray) |

## 8. Понимание вывода программы

### Строки запуска потоков

```
[LAUNCH] Поток  0 | start=00:00:00 clean=00:00:04 → output_seg0000.mkv
```

- `start` — начало сегмента в исходном видео
- `clean` — длительность «чистой» части, которая попадёт в выход

### Строки завершения сегмента

```
[SEG 00] dec=163 enc=267 dur=4.450s → output_seg0000.mkv
```

- `dec=163` — декодировано 163 кадра исходного видео
- `enc=267` — закодировано 267 кадров после интерполяции
- `dur=4.450s` — реальная длительность сегмента

Мультипликатор `267/163 ≈ 1.64`. Ожидаемый теоретический: `60/24 = 2.5×`. Разница объясняется тем, что на короткие сегменты minterpolate тратит часть кадров на «разгон» алгоритма. На длинных видео (фильм) мультипликатор будет близок к теоретическому.

### Итоговая статистика

```
Кадров вход:    745
Кадров выход:   1077
Мультипликатор: ×1.45
Ускорение:      ×0.05 к реальному
```

- **Ускорение** — отношение длительности видео к времени обработки. Значение < 1 означает что обработка медленнее реального времени — это нормально для `mci`+`slow`.

### Прогресс-бар

```
[DONE  1/4] Сег.00 | dec=163 enc=267 dur=4.45s | 7 МБ | ETA 00:13:45  [██████████░░...]
```

ETA рассчитывается по среднему времени завершённых сегментов — может быть неточным если сегменты разной сложности.

## 9. Рекомендации и ограничения

### Когда minterpolate даёт хорошие результаты

- Кино и сериалы с плавным движением камеры
- Документальное видео
- Видеоигры с фиксированной камерой
- Природные съёмки, пейзажи

### Когда minterpolate может давать артефакты

- **Резкие монтажные склейки** — фильтр «размазывает» переход между сценами. Обнаружение смены сцены `scd=fdiff` частично решает это, но не во всех случаях.
- **Текст на экране** — буквы могут «плыть» при движении камеры
- **Очень быстрое движение** (боевики, экшн) — векторы движения могут ошибаться
- **Аниме** — нарочито низкий FPS (8–12) является стилистическим выбором; интерполяция меняет ощущение от анимации

### Дисковое пространство

Временные сегменты занимают дополнительное место. Для 2-часового фильма:
- Входной файл: ~8–15 ГБ (зависит от битрейта)
- Временные сегменты: примерно ×2 от размера выходного файла
- Выходной файл: примерно ×1.5–2 от входного (больше кадров при том же CRF)

Убедитесь что на разделе `--temp-dir` достаточно свободного места.

### Выбор целевого FPS

| Исходный FPS | Рекомендуемые целевые FPS |
|---|---|
| 24 | 48, 60, 120 |
| 25 | 50, 75, 100 |
| 30 | 60, 90, 120 |
| 50 | 100 |
| 60 | 120 |

Кратные значения дают наиболее чистый результат, так как каждый оригинальный кадр точно кратен числу интерполированных. Некратные значения (например, 24→90) работают, но мinterpolate вынужден создавать кадры в нецелых позициях.
